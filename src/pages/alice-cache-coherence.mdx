# Alice ThrOugh the SnOOping Bus

## a Wonderland of Cache Coherence

Welcome to our virtual exhibit! Explore how multi-core processors maintain data consistency through the **Snooping Bus** and **MESI Protocol** that's visualized through the fractured psyche of American McGee's **Alice**.

In this interactive simulation, we are presented with two CPU cores: Classic Alice (Core 0) and Hysteria Alice (Core 1). They share a single block of main memory, in this case we call it the **Rabbit Hole** which serves as the brain of the two personalities. Each core has its own private cache (a *Memory Pocket*) where it stores copies of data from main memory.

### the Metaphor

| Alice Concept | Hardware Concept |
| :--- | :--- |

| Hysteria Alice (Core 1) | CPU Core 1 Cache |

### MESI Protocol States
We chose to present the MESI protocol because it is the industry standard in modern CPUs such as Intel.

| State | Color | Meaning |
| :--- | :--- | :--- |
| **M**odified | Purple | Data has been changed locally and is now "dirty." Only the core that modified the data holds this state. |
| **E**xclusive | White | Only one core has a copy of the data. It matches Main Memory. |
| **S**hared | Gold | Multiple cores have copies of the data. All copies match Main Memory. |
| **I**nvalid | Grey | One core has modified the data so now other cores with the same data must be discarded. |

Beyond the protocol, we also demonstrate two critical cache policies:

*Write-Invalidate* — When a core writes to a cache line in the Shared (S) state, it broadcasts a request on the Snooping Bus. All other cores that hold a copy of that line must invalidate their copies (transition to Invalid/I), ensuring only one core has the latest version. This prevents the "lost update" problem where two cores might independently modify stale data.

*Write-Back* — When a core modifies data in its cache (entering the Modified/M state), it does not immediately write that change back to main memory. Instead, the data stays "dirty" in the cache until explicitly written back. This improves performance by reducing slow memory traffic, but requires careful coherence management. In our exhibit, you must click "Embrace Sanity" to trigger a Writeback operation, flushing the modified data from cache back to RAM.

### How the Simulator Works

1. **Select a data** in the **Rabbit Hole (RAM)**. 
2. **Select a memory slot** in either **Classic Alice (Core 0)** or **Hysteria Alice (Core 1)**. Each slot represents a cache line stored in that core's cache.
3. Click **"Remember"** to perform a **read operation (PrRd)**. If the selected data is not already in the cache, the core retrieves it from **Main Memory** through the Snooping Bus, updating its MESI state accordingly.
4. Click **"Corrupt"** to perform a **write operation (BusRdX)**. This grants the selected core ownership of the cache line while forcing the other core to **invalidate** its copy, maintaining cache coherence.
5. Click **"Embrace Sanity"** to perform a **writeback (WB)**. Any modified ("dirty") data is written back to **Main Memory**, allowing both memory and the cache to become synchronized again.

Observe the **Looking-Glass Bus**, the **Explanation Panel**, and the **MESI state indicators** to see how each operation affects both cores and Main Memory in real time.
